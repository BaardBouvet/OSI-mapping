# ADR 0001: Remove `transform_type` from mapping spec

- Status: Accepted
- Date: 2026-03-07

## Context

The mapping spec originally included a `transform_type` field in each `field_mapping` entry (for example: `rename`, `type_cast`, `conditional`).

At the same time, each mapping already carries:

- `source_expression` (OSI-style dialect expression)
- optional `reverse_expression`
- `source_columns`

This means the transform behavior is already fully represented by executable expressions.

## Problem

`transform_type` duplicates information that is already present in expressions and introduces consistency risk:

- A mapping expression can change while `transform_type` is not updated.
- Validation can pass while metadata is semantically wrong.
- Authors must make an extra manual classification decision for each field.

For a minimal and reliable core spec, duplicated metadata should be avoided unless it is required for execution semantics.

## Decision

Remove `transform_type` from:

1. `specs/osi-mapping-schema.json`
2. Example mapping files:
   - `examples/minimal/erp_to_company_mapping.yaml`
   - `examples/minimal/openapi_to_company_mapping.yaml`

`field_mapping` classification is now inferred from expression content by tooling when needed.

## Consequences

### Positive

- Simpler authoring experience (fewer required/optional fields).
- No stale classification metadata.
- Cleaner minimal spec focused on executable mapping logic.

### Negative

- UI/tooling that wants transform categories must infer them from expressions.
- Inference can be heuristic and dialect-dependent.

## Alternatives considered

### Keep `transform_type` as-is

Rejected due to metadata drift risk and unnecessary duplication.

### Keep `transform_type` but make it computed-only

Possible in tooling, but not persisted in source files. This effectively converges to the chosen approach for the spec itself.

### Keep `transform_type` as strictly validated against parsed SQL

Too complex for the minimal spec and adds parser/dialect coupling to validation.

## Follow-up

If product requirements later need explicit classification (for UX, governance, or reporting), re-introduce it as optional derived metadata generated by tooling, not hand-authored by default.
