# ADR 0004: Remove unmapped field lists from mapping spec

- Status: Accepted
- Date: 2026-03-07

## Context

The mapping spec included two optional arrays on each mapping:

- `unmapped_source_fields` — source fields intentionally excluded from the mapping
- `unmapped_target_fields` — target fields with no source equivalent

Each entry had a `name` and optional `reason`.

## Problem

These lists are computable and stale-prone:

- **Computable**: Given the source model's fields and the mapping's `target_field` entries, both lists can be derived by set difference.
- **Stale-prone**: Adding a field to the source or target model without updating the unmapped lists creates silent inconsistency.
- **Noise**: In practice, minimal mappings end up with empty arrays (`[]`) that add clutter.

This follows the same principle established in ADR 0001 (transform_type) and ADR 0002 (source_columns): hand-authored metadata that duplicates computable information should be generated by tooling, not maintained by authors.

## Decision

Remove from the spec:

1. `unmapped_source_fields` property from `mapping`
2. `unmapped_target_fields` property from `mapping`
3. `unmapped_field` definition (no longer referenced)

Unmapped field analysis becomes a tooling concern — a validation or reporting command that compares source/target models against mapping coverage.

## Consequences

### Positive

- Leaner mapping files focused purely on field transforms.
- No stale documentation to maintain.
- Consistent with the spec's direction of keeping authored files minimal.

### Negative

- Mapping files no longer self-document intentional exclusions and their reasons.
- Tooling must have access to both source and target models to compute coverage gaps.

## Alternatives considered

### Keep as optional documentation

Rejected — optional fields that go stale provide false confidence. If reasons for exclusions are important, they belong in comments or a separate coverage report.

### Move to a generated coverage report file

Viable as a future extension. A CLI tool could produce a `mapping-coverage.yaml` report with unmapped fields, reasons, and warnings.
